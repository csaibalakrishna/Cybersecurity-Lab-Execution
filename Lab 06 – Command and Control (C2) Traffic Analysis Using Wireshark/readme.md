# Lab 6 – Command and Control (C2) Traffic Analysis Using Wireshark

## Overview

This lab demonstrates how **Wireshark** can be used to analyze network traffic generated by a simulated malware sample in order to identify potential **Command and Control (C2)** communications.

A Windows executable was executed inside an isolated environment using Wine. The executable established an outbound TCP connection to a simulated C2 server and transmitted a beacon message. The generated network traffic was captured and analyzed using Wireshark to identify Indicators of Compromise (IOCs), communication patterns, and suspicious network activity.

This exercise simulates one of the most common tasks performed by SOC analysts during malware investigations and incident response.

---

# Objectives

- Capture network traffic generated by a suspicious executable.
- Identify outbound C2 communication.
- Analyze TCP sessions using Wireshark.
- Extract Indicators of Compromise (IOCs).
- Inspect network payloads.
- Understand malware beaconing behavior.

---

# Background

Modern malware rarely operates in isolation. After compromising a system, it commonly establishes communication with an external **Command and Control (C2)** server.

The C2 infrastructure enables attackers to:

- Receive commands
- Upload stolen information
- Download additional malware
- Maintain persistence
- Monitor infected systems

Network traffic analysis is therefore an essential component of malware investigations.

Wireshark enables analysts to inspect captured packets and reconstruct network conversations without modifying the captured evidence.

# Malware Simulation

A Windows executable was compiled to simulate outbound Command and Control communication.

The executable performs the following operations:

- Initializes Windows networking
- Creates a TCP socket
- Connects to a predefined server
- Sends a beacon message
- Terminates the connection

The communication is intentionally simple to demonstrate network analysis techniques in a controlled environment.

---

# Preparing the C2 Listener

A Netcat listener was configured on the simulated C2 server.

```bash
nc -lvnp 4444
```

The listener waits for incoming TCP connections from the executable.

---

# Capturing Network Traffic

Wireshark was started on the monitoring system before executing the binary.

Traffic capture was performed on the Host-Only network interface to observe communication between the infected host and the simulated C2 server.

---

# Executing the Malware Sample

The Windows executable was executed using Wine.

```bash
wine sample.exe
```

The executable silently:

- Created an outbound TCP connection
- Contacted the simulated C2 server
- Transmitted a beacon message

---

# Wireshark Analysis

## Display TCP Traffic

```text
tcp
```

Displays all TCP packets.

---

## Filter C2 Communication

```text
ip.addr == 192.168.56.101
```

Displays communication involving the simulated C2 server.

---

## Filter Suspicious Port

```text
tcp.port == 4444
```

Displays traffic using the predefined C2 communication port.

---

## Inspect TCP Stream

Using **Follow → TCP Stream**, the transmitted payload can be reconstructed.

Observed payload:

```text
beacon:system_online
```

This confirms successful outbound communication between the executable and the simulated C2 server.

The captured traffic revealed several characteristics commonly associated with malware beaconing.

Observed behavior:

- Successful TCP session establishment

# Analysis & Observations

The executable successfully established a connection to the simulated Command and Control server.

Inspection of the TCP stream revealed the beacon message:

```text
beacon:system_online
```

This behavior indicates that the executable is capable of notifying a remote server of its presence after execution.

Although simplified for educational purposes, this communication pattern closely resembles the initial beacon phase commonly observed in real malware families before receiving additional instructions.

# SOC Analyst Perspective

One of the primary responsibilities of a SOC analyst is identifying suspicious outbound network communication.

Rather than relying solely on endpoint alerts, analysts examine network traffic to determine:

- Whether an endpoint is communicating with unauthorized hosts.
- Whether beaconing behavior exists.
- Which systems are affected.
- What information is being transmitted.
- Whether data exfiltration has occurred.

Wireshark enables analysts to reconstruct complete TCP conversations and validate alerts generated by IDS, firewalls, or SIEM platforms.

Understanding C2 traffic is fundamental to malware investigations and network forensics.
